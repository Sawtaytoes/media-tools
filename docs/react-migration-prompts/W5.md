#### Boolean function-vs-variable naming collisions

When a function returning a boolean and a variable holding its return value would naturally share the same `is`/`has` name, the ESLint naming-convention rule makes them collide. The fix is to **prefix the function** with a verb of action (`get`/`check`/`compute`/`determine`) so the variable can take the natural name:

```ts
// ❌ Before: function and variable both want the same name; variable gets a worse name
const isFake = isFakeRequest(context)

// ✅ After: function takes the action-verb form, variable takes the natural boolean name
const isFakeRequest = getIsFakeRequest(context)
```

This pattern is **not enforceable by ESLint** — `@typescript-eslint/naming-convention` will flag `getIsFakeRequest` as a boolean-returning function that doesn't start with `is`/`has`, which is wrong for *this* shape. Disable the rule for boolean-returning functions, OR add a per-function override comment, OR keep the rule but accept the local override pattern. Make the decision once for the codebase and document it.

Hunt by grepping for the shape `is<X> = is<X>(` and `has<X> = has<X>(` — each hit is a collision to fix. Commit message: `refactor: rename boolean-returning functions to get/check prefix (avoid collision with variable name)`.

#### Mutation via `let` + subscription callbacks → `lastValueFrom`

AGENTS.md rule #2 ("`const` only. No `var`. No `let` mutation.") gets violated in tests that subscribe to Observables to detect completion. The pattern looks like:

```ts
// ❌ Before: let + mutation inside a subscribe callback
let hasCompleted = false
subject.subscribe({
  complete: () => {
    hasCompleted = true
  },
})

completeSubject(job.id)

expect(hasCompleted).toBe(true)
expect(getSubject(job.id)).toBeUndefined()
```

The mutation is a workaround for the callback-based subscription — the test really wants to *await* the observable's completion. RxJS's `lastValueFrom` does exactly that:

```ts
// ✅ After: const + await; no mutation
import { lastValueFrom } from "rxjs"

completeSubject(job.id)
await lastValueFrom(subject, { defaultValue: null })

expect(getSubject(job.id)).toBeUndefined()
```

`lastValueFrom(observable, { defaultValue })` returns a Promise that resolves with the last emitted value when the Observable completes. Pass `defaultValue` when the Observable may complete without emitting (otherwise `lastValueFrom` throws `EmptyError`). The `expect(hasCompleted).toBe(true)` line disappears — the `await` is the completion assertion.

If the test needs to assert on emitted values, collect them with `toArray()`:

```ts
import { lastValueFrom } from "rxjs"
import { toArray } from "rxjs/operators"

const emitted = await lastValueFrom(subject.pipe(toArray()))
expect(emitted).toEqual([/* explicit values, no snapshots */])
```

Hunt by grepping for the shape `let .* = false` followed by `.subscribe({` within the next 5 lines — each hit is a candidate. Verify each: some `let` + subscribe combinations are genuinely necessary (e.g., when the test needs to assert mid-stream behavior, not completion). Don't blindly rewrite — read the test intent first.

Commit message: `refactor(tests): replace let+subscribe mutation with lastValueFrom`. Group fixes by file/area; don't single-commit each rewrite.

#### One component per file

Multi-component files explode context and make reading harder. **Every React component (any function returning JSX) lives in its own file**, even if it's a "small internal helper." This is stricter than typical React conventions — the goal is that any future AI session loading a file pays context for one component, not three.

Known offender from W1: [packages/web/src/components/RenderFields/RenderFields.tsx](../../packages/web/src/components/RenderFields/RenderFields.tsx) declares `TodoField`, `FieldDispatcher`, AND `RenderFields` in one 250+ line file. Refactor to:

```text
RenderFields/
├── RenderFields.tsx          # main export only
├── RenderFields.test.tsx
├── RenderFields.stories.tsx
├── RenderFields.mdx
├── FieldDispatcher.tsx       # extracted
└── FieldDispatcher.test.tsx  # if it had tests; otherwise covered by RenderFields tests
```

(`TodoField` may have been deleted by Phase 2's end — verify before extracting; if it still exists, it gets its own file too, even though it's about to disappear.)

Hunt across the whole `packages/web/src/` tree:

```bash
# Files with more than one top-level component (arrow-function form)
yarn dlx ripgrep --type=tsx -l '^(export )?const [A-Z][a-zA-Z]+ = \(' packages/web/src \
  | while read file; do
      count=$(grep -cE '^(export )?const [A-Z][a-zA-Z]+ = \(' "$file")
      [ "$count" -gt 1 ] && echo "$file ($count components)"
    done
```

(Adjust the regex if a worker used `function` declarations or `React.FC`-style components — the migration plan says arrow functions, but check.)

**Non-component helpers** (utility functions, type definitions, constants) MAY stay co-located. The rule is "one component per file," not "one symbol per file."

### Stream 3 — Accessibility pass

Every new field component gets a focused a11y review:

- **Form semantics:** every input has an associated `<label>` (FieldLabel already handles this, but verify `htmlFor` / `aria-labelledby` is wired correctly).
- **Required state:** `aria-required="true"` for `field.required`.
- **Error state:** if a field surfaces a validation error (especially JsonField after Stream 1), the message must be linked via `aria-describedby` and the input must have `aria-invalid="true"`.
- **Keyboard:** every interactive element reachable via Tab; pickers (EnumPicker, LinkPicker, StepOutputPicker) closeable via Escape; focus returns to the trigger after close.
- **ARIA roles:** custom pickers use `role="listbox"` + `role="option"` correctly; combobox patterns where appropriate.
- **Color contrast:** check against WCAG AA (4.5:1 for body text, 3:1 for large text). Tailwind v4 makes this auditable from the class names.

For each a11y fix:

- Land as its own commit so the diff is small.
- Add a test that asserts the a11y property (e.g., `expect(input).toHaveAttribute("aria-required", "true")`).
- Commit message: `feat(a11y): <component> — <what improved>`.
