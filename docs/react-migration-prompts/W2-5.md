## What W2D already discovered (don't re-derive)

W2D's escalation note lists what makes this port non-mechanical. Verbatim from the checklist:

- **`openDetailsKeys` is a module-level `Set`** — port to a Jotai atom (e.g., `dslRulesOpenDetailsAtom`). Keep the same semantics (track which collapsed rule sections are expanded).
- **Bridge pattern inversion:** the legacy code reads `window.mediaTools` and mutates it. You read Jotai atoms and dispatch via `useBuilderActions()` setters. Every `window.mediaTools.*` access in the legacy becomes a `useAtomValue` or `useSetAtom` in the React port.
- **Eight state-shape decisions:**
  1. **Rules union type** — what's the discriminated TS union for a rule? (probably tagged by `kind`)
  2. **Predicates map** — what predicates exist, and how are they keyed?
  3. **`when`/`applyIf` discriminated unions** — are these one type with a `kind` tag, or two separate types?
  4. **`computeFrom` ops chain** — array of operations? Linked list? Pipeline struct?
  5. **Fields map** — keyed by what?
  6. **`scaleResolution` struct** — keep as nested object or flatten?
  7. **`openDetailsKeys`** — atom shape decided above
  8. **One more** — read W2D's escalation; the 8th is in the legacy `state.js` and W2D couldn't compress further

  For each, pick the option that minimizes runtime branching (the user prefers TypeScript discriminated unions over bag-of-optional-fields). Document each choice in the commit message that lands the type.
- **~10 mutation functions** that the legacy invokes via string-interpolated `onclick` handlers — port each to a React handler that calls a Jotai atom setter via `useBuilderActions()`. The mutation names are in `rule-crud.js`.
- **Two commit modes:**
  - Debounced URL sync — when the user edits an inline value (e.g., a text field), debounce ~300ms then update YAML.
  - Immediate re-render — when the user toggles structural state (e.g., adds a rule, deletes a rule, expands a section), commit immediately.
  W2D's escalation doesn't mention exact debounce values — read `state.js` carefully for the timer values and preserve them.
