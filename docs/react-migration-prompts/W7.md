## The migration

### Step 1 — Add @dnd-kit deps, remove sortablejs

```bash
yarn workspace @media-tools/web add @dnd-kit/core @dnd-kit/sortable @dnd-kit/utilities
yarn workspace @media-tools/web remove sortablejs @types/sortablejs
```

`@dnd-kit/utilities` brings the `CSS` transform helper that goes on draggable element styles. The other two are core + the sortable preset (which is what wraps a vertical list of items).

### Step 2 — Rewrite useDragAndDrop (or replace with composition)

The current hook returns refs and wires SortableJS imperatively. With @dnd-kit, the pattern flips inside-out:

- The **container** uses `<DndContext>` (provides drag state) + `<SortableContext>` (knows about the ordered items).
- Each **draggable item** calls `useSortable({ id })` and applies returned `attributes`, `listeners`, `setNodeRef`, and a CSS transform.
- The container handles `onDragEnd` events by emitting an action atom dispatch (e.g., `moveStepAtom`).

Two implementation patterns to choose:

**(a) Compose dnd-kit directly in the consumers (StepCard, GroupCard).** Drop `useDragAndDrop`. Each consumer is responsible for `<DndContext>` / `<SortableContext>` / `useSortable`. Cleaner long-term but spreads the dnd-kit API across multiple files.

**(b) Keep a `useDragAndDrop` shim with a new internal implementation.** Same external API (`{ containerRef, itemRefs }` or similar) but powered by dnd-kit underneath. Minimal blast radius on call sites; can be refactored to pattern (a) later.

**Recommended: (a).** dnd-kit is meant to be composed at the consumer; wrapping it in a hook adds indirection without value. The current `useDragAndDrop` is a shim FOR SortableJS specifically — it doesn't carry over.

### Step 3 — Wire keyboard + a11y

@dnd-kit ships keyboard support out of the box: arrow keys move focused items, Space picks up / drops, Escape cancels. Test this in the dev server — make sure focusable handles (drag-handle icon) are reachable via Tab.

Add a `<DragOverlay>` to render the floating preview of the dragged item. Without it, dragging looks abrupt; with it, the preview follows the cursor smoothly.

### Step 4 — Update tests

Existing tests that mock SortableJS (search for `vi.mock('sortablejs')` or similar) need rewriting:

- For unit/component tests: use `@dnd-kit/core`'s test utilities, or assert on the result of an `onDragEnd` callback firing.
- For e2e (Playwright) — W6 likely uses `data-drag-handle` or similar selectors. @dnd-kit emits its own DOM attributes (`data-dnd-kit-...`); update spec selectors accordingly. Coordinate with W6's branch (if W6 hasn't merged yet) or rebase W6's specs after this lands.

### Step 5 — Verify drag UX matches

Run the dev server, manually:
- Drag a step from position 1 to position 3 — order updates, YAML shows new order
- Drag a step into a parallel group — gets inserted at the drop position
- Drag a step OUT of a group back to top level — leaves the group correctly
- Keyboard: focus a drag handle, press Space, arrow down, press Space — same reorder
- Verify the animation: dnd-kit's default transition is ~150ms; tune `transition` prop on `useSortable` if it feels too fast/slow
